## Quick Start

```bash
# Install dependencies
pnpm install

# Build Lambda packages
pnpm run build

# Run tests
pnpm run test

# Deploy to sandbox
cd infrastructure/sandbox/eu-west-2/dice-api
aws-vault exec sandbox -- terragrunt apply
```

## API Endpoints

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/roll` | None | Roll a dice (1-6) |
| GET | `/stats` | None | Get aggregated statistics |
| DELETE | `/reset` | `x-api-key` header | Reset all rolls and stats |

### Try it now

```bash
API_URL="https://ixvk4ixi35.execute-api.eu-west-2.amazonaws.com"

# Check stats (updates asynchronously — wait 1-2 seconds after rolling)
```

## ⚠️ Known Simplifications

> **Do not copy these patterns into production code.** This project is a teaching tool and deliberately takes shortcuts to keep the code approachable. The following are intentional simplifications that would need addressing in a real service:

| # | Location | Simplification | Production Approach |
|---|----------|---------------|---------------------|
| 1 | `src/api/roll.ts` | `Math.random()` for dice roll | Use `crypto.randomInt()` — `Math.random()` is not cryptographically secure |
| 2 | `src/api/authoriser.ts` | `===` string comparison for API key | Use `crypto.timingSafeEqual()` to prevent timing side-channel attacks |
| 3 | `src/api/authoriser.ts` | Secret cached indefinitely (no TTL) | Add a TTL-based cache or use the Secrets Manager Lambda extension so rotated secrets are picked up without a cold start |
| 4 | All handlers | `process.env.VAR!` with no validation | Validate environment variables at cold-start and fail fast with a clear error message |
| 5 | `src/api/reset.ts` | Full table scan + batch delete to clear data | Use DynamoDB TTL, recreate the table, or avoid exposing a "delete everything" endpoint |
| 6 | `src/api/reset.ts` | No backoff on `UnprocessedItems` retry loop | Add jittered exponential backoff to avoid hammering DynamoDB when throttled |
| 7 | `infrastructure/.../sqs.tf` | SNS→SQS without raw message delivery | Set `raw_message_delivery = true` on the subscription to avoid double JSON parsing |
| 8 | `src/api/roll.ts` | No input validation on the endpoint | Explicitly validate and reject malformed or unexpected request payloads |
| 9 | DynamoDB schema | Single table, no GSIs, minimal access patterns | Separate concerns with dedicated tables or GSIs as access patterns grow |

## Learning Path

1. Run `pnpm run test` — see what's being tested
2. Read `src/api/roll.ts` — the simplest Lambda (DynamoDB write + SNS publish)
3. Read `src/processors/stats-processor.ts` — consuming SQS events
4. Read `src/utils/update-stats.ts` — DynamoDB update expressions
5. Read `docs/guide.md` — full context on services and wiring
6. Read `infrastructure/modules/dice-api/lambda-roll.tf` — how Terraform creates and wires a Lambda
7. Deploy to your sandbox with `terragrunt apply`
8. Call the API and watch CloudWatch Logs in the AWS Console

For a deeper dive — including toggling between the two event paths (direct SNS publish vs. DynamoDB Streams) and other hands-on activities — see the [Beginner's Guide](docs/guide.md).

## Extension Points

Ideas for extending the service, in keeping with the existing features and AWS patterns. These are intentionally high-level — each can be expanded into a detailed spec.

- **Improve some of the simplifications**
    - Remove the use of `Math.random()` in `src/api/roll.ts` in favour of something cryptographically secure (e.g. `crypto.randomInt()`).
    - Replace the string comparison in `src/api/authoriser.ts` with something more appropriate (e.g. `crypto.timingSafeEqual()`).
    - Use AWS Powertools to fetch the API key secret so it's not cached indefinitely.
    - Update the SNS subscription to use raw message delivery.
- **Add an additional SNS topic** that only receives a notification when there is a high roll (e.g. a 6 on a D6).
- **Create alarms** to ensure the system is working correctly.
- **Protect against duplicate events** from the stream processor.
    - Hint: look at AWS Powertools + idempotency.
- **Create a snapshot of the stats** every hour, on the hour.
- **Add a query param** that allows specifying the die type (e.g. `?d=20`).
    - Hint: think about the downstream impacts for things like the stats endpoint.
